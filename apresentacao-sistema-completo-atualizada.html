Backend 4/10

Endpoints de produtos

  • `GET /api/produtos` lista produtos.
  • `POST /api/produtos` cria produto.
  • `PUT /api/produtos/{id}` atualiza produto.
  • `DELETE /api/produtos/{id}` remove produto.
{ "nome": "Arroz 5kg", "preco": 24.90, "quantidade": 20, "unidade": "UN" }
Backend 5/10

Endpoints de movimentações

  • `GET /api/movimentacoes` lista o histórico.
  • `POST /api/movimentacoes` registra entrada ou saída.
  • Entrada soma quantidade no produto.
  • Saída subtrai quantidade se houver saldo.
{ "produto_id": 1, "tipo": "SAÍDA", "quantidade": 3 }
Backend 6/10

Regra de estoque

A principal regra do backend é impedir que uma saída deixe o estoque negativo.

  • Busca o produto no banco.
  • Confere a empresa atual.
  • Compara quantidade disponivel e quantidade solicitada.
  • Atualiza produto e grava histórico.

Backend 7/10

Contexto de empresa

O backend separa dados por empresa usando o JWT gerado no login.

  • `empresaId` identifica a empresa no token.
  • `usuarioId` identifica o usuario logado.
  • `role` define permissões de acesso.
  • Headers manuais ficam apenas para testes tecnicos.
Authorization: Bearer <jwt> { empresaId, usuarioId, role }

  • Somente o dono cadastra empresas clientes.
  • Cada empresa recebe um administrador inicial.
  • Usuários de empresas entram por `/login` com JWT.

Banco 1/10

Banco de Dados

O MySQL guarda os dados persistentes do ControlX e define os relacionamentos principais.

Banco 2/10

Banco utilizado

  • SGBD: MySQL.
  • Banco local padrao: `estoque_inteligente`.
  • Conexao configurada em `application.properties`.
  • Spring Data JPA faz o mapeamento Java -> SQL.
  • Arquivos base: `schema.sql` e `seed.sql`.
spring.datasource.url=jdbc:mysql://localhost:3306/estoque_inteligente spring.datasource.username=root spring.datasource.password=
Banco 3/10

Tabela empresas

  • Representa uma organização dona dos dados.
  • Tem `id`, `nome`, `cnpj`, `tema_empresa_id` e `created_at`.
  • `cnpj` possui chave única.
  • Serve como raiz do modelo multiempresa.
empresas
id BIGINT PK
nome VARCHAR(150)
cnpj VARCHAR(20) UNIQUE
tema_empresa_id VARCHAR(40)
created_at DATETIME
Banco 4/10

Tabela usuários

  • Cada usuário pertence a uma empresa.
  • Guarda nome, email e senha criptografada.
  • `email` e único.
  • `role` separa ADMIN, OPERADOR e VISUALIZADOR.
  • Também armazena permissões, `ativo` e `session_version`.
usuários
empresa_id FK

Banco 5/10

Tabela produtos

  • Guarda os itens do estoque.
  • Tem nome, preço, quantidade e unidade.
  • Produto pertence a uma empresa.
  • Tem `created_at` e `updated_at` para auditoria.
  • Nome não pode repetir dentro da mesma empresa.

Banco 7/10

Onde liga no sistema

  • `application.properties` define a conexao com o MySQL.
  • As classes `model` mapeiam as tabelas com JPA.
  • Os `repositories` executam consultas e persistência.
  • Os `services` chamam `save`, `findBy...` e `delete`.
Frontend -> /api -> Controller -> Service -> Repository -> MySQL
Banco 8/10

Relacionamentos

O modelo relacional garante isolamento por empresa e histórico por produto.

  • Uma empresa tem muitos usuários.
  • Uma empresa tem muitos produtos.
  • Uma empresa tem muitas movimentações.
  • Um produto tem muitas movimentações.
empresas 1 -> N usuários empresas 1 -> N produtos empresas 1 -> N movimentações produtos 1 -> N movimentações

Chaves e integridade

  • Chaves únicas evitam duplicidade indevida.
  • Indices por empresa aceleram filtros multiempresa.
  • Índices por nome e data ajudam busca e relatórios.
